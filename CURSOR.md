# Cursor 하네스 (프로젝트 규약)

이 문서는 **Cursor**에서 돌아가는 에이전트·챗·인라인 편집이 이 저장소를 다룰 때 따를 **하네스**(입력·출력을 제한하는 규약)다. 철학은 [Andrej Karpathy](https://x.com/karpathy)가 널리 제안한 **네 가지 원칙**(구현 전 사고 → 단순성 → 수술적 수정 → 목표 검증)과 같다.

**역할 분담**

| 파일 | 역할 |
|------|------|
| `.cursorrules` | 세션마다 주입되는 **짧은 강제 규약** |
| `CLAUDE.md` | 같은 철학의 **풀 텍스트**·예시·점검 항목 |
| `CURSOR.md` (본 문서) | Cursor 맥락에서의 **하네스 설계 의도**와 운영 원칙 |

프로젝트별 규칙이 있으면 위 파일들과 **모순 없이 병합**한다. 충돌하면 **더 좁은 범위·더 강한 검증** 쪽을 택한다.

> **트레이드오프:** 속도보다 **신중함**. 사소한 작업은 완화할 수 있으나, 아래 **금지·필수**는 기본값으로 둔다.

---

## 하네스가 막는 실패

- **침묵 가정:** 스펙을 추측으로 채우지 않는다.
- **범위 팽창:** “겸사겸사” 리팩터·기능 추가를 하지 않는다.
- **검증 공백:** “돌아가는 것 같다”로 끝내지 않는다.

---

## 1. 구현 전 사고 (Think Before Coding)

**가정하지 않는다.** 모호함을 덮지 않고, 트레이드오프는 말로 밝힌다.

- 가정은 항상 **문장으로** 낸다. 불확실하면 **질문**한 뒤에 손을 댄다.
- 해석이 여러 개면 **하나를 몰래 고르지 말고** 대안을 나열한다.
- 더 단순한 해법이 있으면 **제안**한다. 이유가 있으면 요청에 **정중히 반대**해도 된다.
- 요구가 불명확하면 **멈추고**, 헷갈리는 조각을 **이름 붙여** 질문한다.

---

## 2. 단순성 우선 (Simplicity First)

- 요청을 만족하는 **최소 변경·최소 코드**만 쓴다.
- 요청받지 않은 기능·설정·추상화·일회용 레이어를 두지 않는다.
- 일어나지 않을 상황을 가정한 방어·예외 처리를 쌓지 않는다.
- **“시니어가 과하게 복잡하다고 할까?”**에 예면 단순화한다.

---

## 3. 정밀한 수정 (Surgical Changes)

- **요청과 직접 연결된** 경로·줄만 바꾼다. 옆줄 포맷·주석·스타일 “정리”로 범위를 넓히지 않는다.
- 망가지지 않은 코드는 리팩터하지 않는다. **기존 스타일**을 따른다.
- 작업과 무관한 데드 코드는 **알리기만** 하고 임의 삭제하지 않는다.
- **내 수정**으로 불필요해진 import·변수·함수만 제거한다. 원래부터 있던 데드 코드는 요청 없이 건드리지 않는다.

**변경 라인 규칙:** diff의 각 줄이 사용자 요청과 **한 줄로 설명** 가능해야 한다.

---

## 4. 목표 중심 실행 (Goal-Driven Execution)

- 작업 시작 시 **성공 조건**(무엇이 참이면 끝인지)을 짧게 고정한다.
- “버그 수정·검증·리팩터”는 가능하면 **재현 테스트 또는 동등한 검증**으로 끝을 맞춘다.
- 단계가 많으면 **계획 + 단계마다 검증 방법**을 적고 진행한다.
- 검증 없이 “완료”라고 말하지 않는다.

---

## Cursor에서의 운영 팁

- **Rules for AI**·프로젝트 규칙에 `.cursorrules`가 들어가면, 본 문서는 “왜 이렇게 묶었는지”를 사람과 에이전트가 같이 읽는 **설계 메모**로 쓴다.
- 긴 예시·문장은 `CLAUDE.md`를 우선 참고한다.

---

## 하네스 자가 점검

- 이번 변경이 **요청 범위 안**인가?
- **가정·트레이드오프**를 사용자에게 남겼는가?
- **검증**을 실행했거나, 불가능하면 그 이유를 분명히 했는가?
